# higgsfield-demo

**Scroll-driven cinematic motion websites using the exact Higgsfield workflow from your X bookmarks.**

Grok Imagine (start + end frames) → open-source Higgsfield clone (reference_to_video / Cinema Studio img2vid with the two frames as image references + camera controls) → production-grade scroll-scrubbed 3D site (Next.js + GSAP + Three.js VideoTexture + layered planes + camera dolly).

Pure scroll. No play buttons. The generated photographic content (guy + truck) is the undeniable hero in the 3D scene.

## Why this exists

You asked: "are you 1000% certain that an html file is the best way to handle this? why dont we switch to posting this on vercel under a new project and gh repo called higgsfield demo"

The single-file local HTML (file://) hit hard limits:
- Video duration/metadata unreliable
- currentTime seeking + VideoTexture updates often did nothing visible
- JPG textures for the actual frames (guy + truck) never came through — particles or black planes dominated
- Impossible to verify "does the animation actually play when I scroll" like the real X examples (igloo.inc, bonsaixbt canvas+sticky+progress technique)

A proper deployed Next.js app on Vercel fixes all of that. Real hosting = reliable video, instant textures, proper seeking, camera that actually reveals the content, and a clean URL you can send to clients or use in pitches.

This is the permanent home for the technique.

## The exact workflow (your bookmarks)

1. Ground with real brand kit (photos from site + FB + GBP, voice, reviews).
2. Generate 2–4 grounded cinematic clips via open clone (Anil Matcha Open-Generative-AI Cinema Studio or HyperFrames). Use the two Imagine frames as image references + precise camera language (slow dolly-in 50mm, natural light, subtle crane, aperture, etc.).
3. One prompt to Claude → full scroll-driven website (tall sticky container, rect-based progress, video.currentTime or canvas frame batches, Three.js or WebGL for 3D depth, GSAP ScrollTrigger accents, overlays timed to progress, reduced-motion fallback).
4. Deliver CREDITS + the exact regeneration prompts so the client is never locked in.

Total cost: $0–$12 of your time/electricity. Perceived client value: $4,500–$9,000 one-time (plus optional content factory retainer).

## Running locally

```bash
npm install
npm run dev
```

Open http://localhost:3000

Scroll the tall "LIVE DEMO" area. The 3D scene (canvas) should scrub the transition video while the camera dollies and the layered start/end planes (your actual Imagine frames) move in 3D space. The guy + truck content is the hero — particles are only atmospheric mist in the background.

Click the tiny debug readout in the bottom-left of the canvas area to re-kick the video if needed during dev.

## Deploy (Vercel + GitHub)

1. Create a new GitHub repo named exactly `higgsfield-demo`.
2. In this folder:
   ```bash
   git remote add origin https://github.com/YOURUSER/higgsfield-demo.git
   git branch -M main
   git push -u origin main
   ```
3. Go to Vercel → New Project → Import Git Repository → pick `higgsfield-demo`.
4. Deploy. (GitHub integration means every future push to main auto-deploys — the same "push triggers everything" pattern you use for MoltOS.)

No manual `vercel --prod`. No secrets in the repo.

## Key implementation notes (for when you want to evolve it)

- Tall sticky container (320vh) + rect-based progress calculation = user fully controls the timeline by scrolling (exactly like the bonsaixbt bookmark and igloo.inc examples).
- Three.js r128-era code ported to current three (0.184) using VideoTexture for the MP4 + TextureLoader for the two JPGs + BufferGeometry points for subtle mist + PerspectiveCamera dolly + plane transforms all driven by the same `prog`.
- The video element is hidden; the canvas is the visible output (WebGL). This gives you the 3D camera moves "around" the generated content.
- On real hosting the texture updates and seeking are reliable — this was the entire point of moving off the single HTML file.
- Reduced-motion: we pause the video and can (in future) swap to a static high-quality end frame or Lottie.
- The visible start/end images + full workflow explanation stay on the page so the demo teaches how to replicate (this is the sales tool for StandoutLocal).

Future upgrades you can do in this repo without starting over:
- Pre-extract frames from the MP4 at build time (or a small script) into /public/frames/*.webp and switch the scrub to the exact "canvas + preloaded frame batches + instant drawImage" technique from the bonsaixbt post for 100% flicker-free Apple-style control.
- Add more clips / a small gallery of variations.
- Turn the workflow section into its own /workflow route with copy buttons for the prompts.
- Swap the vanilla Three for @react-three/fiber + drei if you want declarative 3D later.
